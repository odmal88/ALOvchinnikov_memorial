# 09_SOURCE_JSON

## Назначение

`09_SOURCE_JSON` — это контентный слой сайта мемориальной выставки Александра Львовича Овчинникова.

Его задача — отделить тексты и повторяющиеся выставочные данные от HTML-разметки, чтобы:

- редактировать содержание страниц без ручного переписывания шаблонов;
- убрать дублирование дат, адреса и служебных формулировок;
- сделать редактуру и работу Codex предсказуемой;
- иметь один понятный источник правды для сайта.

## Главный принцип

В проекте не должно быть нескольких равноправных источников одного и того же текста.

Правильная логика такая:

- `shared/`, `pages/` и `templates/` — редактируемые источники;
- `drafts/` — рабочие редакторские файлы, не runtime-источник;
- `build/site_texts_ru_master_v3.json` — производный сборочный файл;
- HTML-страницы должны постепенно читать тексты из JSON, а не хранить их напрямую.

## Роли папок

### `shared/`
Общие данные, повторяющиеся на нескольких страницах:
- название проекта;
- официальный подзаголовок выставки;
- даты выставки;
- открытие;
- место проведения;
- город;
- адрес;
- повторяющиеся служебные формулировки.

### `pages/`
Отдельные контентные источники для страниц сайта.
Один файл = одна страница или один крупный смысловой модуль.

### `templates/`
Шаблонные тексты для страниц произведений и QR-материалов.

### `drafts/`
Рабочие редакторские и переходные файлы.
Они не являются runtime-источником и не должны напрямую подключаться к сайту.

### `build/`
Сборочные файлы.
`build/site_texts_ru_master_v3.json` создаётся из `shared/`, `pages/` и `templates/`.
Его не нужно редактировать вручную.

## Источник правды

### Источник правды для редактуры
- `shared/site.json`
- `pages/*.json`
- `templates/*.json`

### Не источник правды
- `drafts/*.json`
- `build/site_texts_ru_master_v3.json`
- старые тексты, оставшиеся в HTML после миграции

## Связь с HTML

Текущая структура сайта исторически хранит много текстов прямо в:
- `pages/*.html`
- `components/*.html`

Целевая модель такая:
- HTML отвечает за разметку и композицию;
- CSS отвечает за стиль;
- JSON отвечает за текст и повторяющиеся данные.

## Какой файл за что отвечает

- `pages/home.json` — главная
- `pages/exhibition.json` — страница выставки
- `pages/about.json` — страница художника
- `pages/collection.json` — коллекция
- `pages/route.json` — маршрут и места
- `pages/linocut.json` — линогравюра
- `pages/virtual.json` — виртуальный зал
- `pages/memory.json` — книга памяти
- `pages/visit.json` — посещение

Важно: даже если в проекте есть `artist_page_copy_for_codex.json`, он считается только рабочим draft-файлом. Основной источник для страницы художника — `pages/about.json`.

## Почему не один giant JSON

Единый master-файл удобен как экспорт, но неудобен как основной ручной источник.

Проблемы одного большого файла:
- шумные diff;
- высокий риск случайно зацепить чужой раздел;
- трудно быстро понять, какой блок относится к какой странице;
- неудобно давать Codex точечные задачи.

Поэтому принята схема:
- редактирование по страницам;
- сборка в один master.

## Сборка

```bash
node 09_SOURCE_JSON/build_master.mjs
```

Скрипт собирает:
- `shared/site.json`
- `pages/*.json`
- `templates/*.json`

в файл:

```text
09_SOURCE_JSON/build/site_texts_ru_master_v3.json
```

## Правила работы

1. Не редактировать `build/site_texts_ru_master_v3.json` вручную.
2. Не подключать `drafts/*.json` напрямую к runtime.
3. Не дублировать один и тот же текст одновременно в page JSON и в HTML без явной причины.
4. Если утверждён новый редакторский текст, переносить его в соответствующий `pages/*.json`.
5. Повторяющиеся выставочные данные менять только в `shared/site.json`.
6. Если для страницы уже есть page-source, именно он считается главным источником правды.
