# ARCHITECTURE

## Назначение

Этот репозиторий содержит статический сайт мемориальной выставки Александра Львовича Овчинникова.

Сайт работает как **статическое приложение с hash-router**:
- HTML отвечает за композицию страниц и DOM-каркас;
- CSS отвечает за визуальный слой;
- JSON отвечает за тексты и повторяющиеся выставочные данные там, где миграция уже выполнена;
- JavaScript связывает всё это в runtime.

Главная цель текущей архитектуры — **отделить контент от шаблонов**, не ломая уже существующий статический сайт.

---

## Общая схема работы

Порядок загрузки приложения:

1. `index.html` подключает `bootstrap.js`.
2. `bootstrap.js` загружает:
   - общие HTML-компоненты (`header`, `mobile-menu`, `footer`);
   - HTML-страницы из `pages/*.html`;
   - `components/home-teasers.html`.
3. После загрузки HTML-каркаса `bootstrap.js` подключает `content-sync.js`.
4. `content-sync.js` загружает целевые JSON-источники и применяет текст в уже существующий DOM.
5. `bootstrap.js` ждёт завершения `window.__contentSyncPromise`.
6. Только после этого подключается `app.js`.
7. `app.js` запускает роутер, интерактив, базу работ, фильтры, 3D-модули и логику страниц.

Коротко:

```text
index.html
  -> bootstrap.js
    -> components/*.html
    -> pages/*.html
    -> components/home-teasers.html
    -> content-sync.js
      -> JSON sync
    -> app.js
      -> router / interactions / worksDB / 3D
```

## Ключевые файлы

### `index.html`
Точка входа приложения.

Содержит:
- базовый shell документа;
- `#component-header`
- `#component-mobile-menu`
- `#pages-root`
- `#component-footer`
- подключение `bootstrap.js`

### `bootstrap.js`
Первичный загрузчик приложения.

Отвечает за:
- fetch HTML-компонентов;
- fetch HTML-страниц;
- вставку `home-teasers`;
- подключение `content-sync.js`;
- ожидание `window.__contentSyncPromise`;
- подключение `app.js`.

### `content-sync.js`
Слой мягкой синхронизации контента из JSON в DOM.

Это **не новый движок рендеринга**, а тонкая прослойка, которая:
- читает JSON;
- находит уже существующие DOM-узлы;
- проставляет в них текст, маршруты и повторяющиеся данные.

Идея:
- не переписывать сайт на framework;
- не удалять HTML-шаблоны;
- постепенно переводить смысловой контент в JSON.

### `app.js`
Основная логика приложения.

Содержит:
- ambient background;
- `worksDB`;
- `categoryLabels`;
- `routeMap`;
- `pageTitles`;
- hash-router;
- логику страниц работ;
- фильтры каталога;
- инициализацию 3D-мастерской;
- mobile menu и навигацию.

## Что является источником правды

### Источник правды для текста
- `09_SOURCE_JSON/shared/site.json`
- `09_SOURCE_JSON/pages/*.json`
- `09_SOURCE_JSON/templates/*.json`

### Не источник правды
- `09_SOURCE_JSON/drafts/*.json`
- `09_SOURCE_JSON/build/site_texts_ru_master_v3.json`
- старые текстовые фрагменты, оставшиеся в `pages/*.html` после миграции

## Что уже подключено к runtime через JSON

На текущем этапе `content-sync.js` читает только:

- `09_SOURCE_JSON/shared/site.json`
- `09_SOURCE_JSON/pages/home.json`
- `09_SOURCE_JSON/pages/exhibition.json`
- `09_SOURCE_JSON/pages/about.json`
- `09_SOURCE_JSON/pages/visit.json`

Это значит:

### Уже управляется через JSON
- Главная
- Выставка
- О художнике
- Посещение
- Повторяющиеся выставочные данные (частично)

### Ещё не полностью переведено на JSON runtime
- Работы / каталог
- Цифровое пространство
- Маршрут и места
- Линогравюра
- Книга памяти
- некоторые вторичные текстовые зоны и внутренние модули

Наличие JSON-файла в `09_SOURCE_JSON/pages/` **не означает автоматически**, что страница уже подключена к runtime.
Подключение происходит только через `content-sync.js`.

---

## Соотношение HTML и JSON

### HTML
Хранит:
- структуру страницы;
- сетку;
- заголовочные области;
- DOM-узлы для будущей подстановки текста;
- layout-компоненты.

### JSON
Хранит:
- редакторский текст;
- повторяющиеся данные выставки;
- page-specific copy;
- шаблонный контент.

### `content-sync.js`
Связывает JSON и HTML.

---

## Текущее правило редактуры

### Если страница уже подключена к JSON runtime
Править нужно **JSON**, а не HTML.

То есть:

- главная -> `09_SOURCE_JSON/pages/home.json`
- выставка -> `09_SOURCE_JSON/pages/exhibition.json`
- художник -> `09_SOURCE_JSON/pages/about.json`
- посещение -> `09_SOURCE_JSON/pages/visit.json`
- общие данные -> `09_SOURCE_JSON/shared/site.json`

### Если страница ещё не подключена к runtime
Нужно сначала проверить:
1. существует ли для неё JSON-файл;
2. есть ли под неё синхронизация в `content-sync.js`;
3. откуда реально берётся текст сейчас — из JSON, HTML или `app.js`.

## Роутинг

Сайт использует hash-router.

Основные маршруты:
- `#/`
- `#/artist`
- `#/exhibition`
- `#/works`
- `#/digital`
- `#/digital/workshop`
- `#/digital/animated`
- `#/visit`

Также поддерживаются work detail routes:

```text
#/works/:slug
```

Они строятся из `worksDB` в `app.js`.

---

## Откуда берутся работы

Каталог работ и отдельные страницы работ сейчас завязаны на:

```js
const worksDB = [...]
```

в `app.js`.

Это значит:
- каталог произведений пока не переведён на JSON content layer полностью;
- изменение карточек работ, slug, описаний и метаданных может по-прежнему потребовать правки `app.js`, если они берутся из `worksDB`.

---

## Повторяющиеся данные выставки

Повторяющиеся данные должны храниться в:

```text
09_SOURCE_JSON/shared/site.json
```

Туда относятся:
- название проекта;
- официальный подзаголовок выставки;
- даты;
- открытие;
- зал;
- город;
- адрес;
- служебные повторяющиеся формулировки.

Если меняются дата выставки, адрес или opening info — править нужно **только этот файл**, а не несколько HTML-шаблонов вручную.

---

## Как работает `content-sync.js`

`content-sync.js` не рендерит страницы с нуля.

Он работает так:
1. fetch JSON;
2. ищет существующие DOM-узлы по селекторам;
3. подставляет:
   - `textContent`
   - `innerHTML`
   - `data-route`
   - `href`

Для выставки используются устойчивые привязки через:

```html
data-sync-section="..."
```

Это предпочтительнее, чем `nth-of-type(...)`.

---

## Когда добавлять новый текстовый блок

Если нужно добавить новую редакторскую зону на страницу, порядок такой:

1. создать или расширить JSON в `09_SOURCE_JSON/pages/...`
2. убедиться, что в HTML есть стабильный DOM-контейнер
3. добавить синхронизацию в `content-sync.js`
4. не делать HTML основным источником текста
5. при необходимости обновить `09_SOURCE_JSON/README.md` и этот `ARCHITECTURE.md`

---

## Когда можно править HTML руками

HTML можно править руками, если задача относится к:
- структуре;
- layout;
- DOM-контейнерам;
- data-атрибутам для sync;
- кнопкам/обёрткам/секциям;
- новой разметке для будущего JSON-контента.

Но **не стоит** использовать HTML как основной источник редакторского текста для уже переведённых страниц.

---

## Когда можно править `app.js`

`app.js` нужно править только если задача относится к:
- роутингу;
- `worksDB`;
- интерактиву;
- 3D-мастерской;
- фильтрам;
- анимациям;
- mobile menu / navigation behavior;
- page activation logic.

Для обычной редакторской правки текста `app.js` трогать не нужно.

---

## Роль `drafts/`

`09_SOURCE_JSON/drafts/` — это рабочая редакторская зона.

Туда можно складывать:
- промежуточные версии текста;
- codex-oriented drafts;
- тексты на обсуждении;
- переходные JSON.

Но:
- `drafts/*.json` не должны быть runtime-источником;
- утверждённый текст переносится в `pages/*.json` или `shared/site.json`.

---

## Роль `build/`

`09_SOURCE_JSON/build/site_texts_ru_master_v3.json` — сборочный файл.

Он нужен:
- для экспорта;
- для контроля структуры;
- для возможного единичного импорта;
- как snapshot всего текстового слоя.

Его **не редактируют вручную**.

---

## Сборка master-файла

Сборка выполняется так:

```bash
node 09_SOURCE_JSON/build_master.mjs
```

Скрипт собирает:
- `shared/site.json`
- `pages/*.json`
- `templates/*.json`

в один файл:

```text
09_SOURCE_JSON/build/site_texts_ru_master_v3.json
```

## Что нужно помнить в новых чатах и новых задачах

Если работа идёт над редактурой страницы, сначала нужно спросить:

1. Эта страница уже подключена к JSON runtime?
2. Какой файл является источником правды?
3. Не пытаюсь ли я вернуть текст обратно в HTML?

Правило по умолчанию:

> Если страница уже мигрирована на `09_SOURCE_JSON`, текст нужно править в JSON, а не в HTML.

Хорошая формула для новых задач:

> В этом проекте источник текста — `09_SOURCE_JSON/shared/site.json` и `09_SOURCE_JSON/pages/*.json`. HTML — каркас. Не переносить утверждённый текст обратно в `pages/*.html`, если нет отдельной технической причины.

---

## Минимальный чек перед merge

Перед merge любой текстовой правки нужно проверить:

- [ ] источник правды — JSON, а не HTML
- [ ] `drafts/` не подключён к runtime
- [ ] повторяющиеся данные не дублируются вручную в нескольких местах
- [ ] если изменён `content-sync.js`, селекторы совпадают с реальным DOM
- [ ] если изменён `bootstrap.js`, порядок инициализации не сломан
- [ ] если правился `pages/exhibition.html` или похожий HTML, не разрушены `data-sync-section` / контейнеры для sync
- [ ] если правились тексты работ — проверено, не живут ли они в `worksDB`

## Текущее состояние проекта

Архитектура сейчас гибридная:

- сайт исторически начался как HTML-first;
- контентный слой вынесен в `09_SOURCE_JSON`;
- для ключевых страниц уже существует runtime sync;
- часть страниц всё ещё остаётся на старой логике;
- переход продолжается постепенно, без полного переписывания фронтенда.

Это осознанное решение:
**мигрировать контент в JSON поэтапно, сохраняя стабильность сайта.**
